# Phase 0: Preflight

## Purpose

Preflight confirms that the workspace, local system, required tools, DNS, and basic target reachability are ready before any assessment traffic is generated by later phases.

## What this phase tests

- Workspace structure and required configuration values.
- Kali/Debian package-health signals that can be checked safely.
- Required command-line tools for enabled phases.
- DNS resolution for the configured target host.
- One low-impact reachability request to the configured target or login URL.

## What this phase does not test

- It does not install missing packages or upgrade the operating system.
- It does not perform vulnerability scanning.
- It does not authenticate to the application.
- It does not crawl, fuzz, brute force, or enumerate broad target ranges.

## Default command

```bash
./phases/00-preflight.sh --workspace assessments/<company>/<target>/<run-id>
```

For approved non-interactive runs:

```bash
./phases/00-preflight.sh --workspace assessments/<company>/<target>/<run-id> --yes
```

## Useful options

- `--yes`: confirms the operator has reviewed scope and wants non-interactive execution.
- `-h`, `--help`: prints usage.

Preflight does not support `--clean` because it should not create large scanner evidence that needs phase reset.

## Profile/depth controls

Preflight is intentionally minimal and is not meaningfully deepened by profiles. It reads the workspace target and profile values so later phases can depend on the same configuration.

## Evidence produced

Evidence is written under:

```text
evidence/phase-0-preflight/
status/phase-0-preflight.status
```

Evidence typically includes preflight notes, tool-check results, DNS output, and the result of the low-impact reachability request.

## Expected results

A healthy run should show:

- Valid workspace config.
- Required tools present.
- DNS resolving the target host.
- Target reachability confirmed with a low-impact HTTP request.
- Warnings, not failures, for optional tools when later phases can tolerate them.

Missing ZAP or `testssl`/`testssl.sh` may be a warning depending on implementation and which later phases are planned. Missing required tools should fail.

## How to interpret findings

Preflight findings are readiness findings, not application vulnerabilities. Treat failures as blockers for later phases until the operator fixes the local environment, workspace configuration, or scope definition.

## Common false positives/noise

- Package health checks may warn when passwordless sudo is unavailable. The operator may need to run the recommended `sudo apt-get check` manually.
- DNS may resolve differently from another network location.
- A reachability request may fail because a WAF, allowlist, VPN, or proxy is required.
- Optional tool warnings do not always block the run.

## Safety and performance notes

- Does not install or upgrade packages.
- May use passwordless sudo for `apt-get check` if available; otherwise it should warn or ask the operator to perform a manual check.
- Sends only a low-impact reachability request.

## Troubleshooting

- Confirm the workspace path is correct.
- Review `config/target.env` for target URL, login URL, profile, and host values.
- Run `./install.sh --check-only` to review dependency status.
- Manually run DNS and a single `curl -I` from the assessment host if network access is unclear.
- If sudo/package checks warn, run the manual package-health command approved for the host.

## When to increase scope/depth

Do not increase scope in preflight. Instead, fix readiness issues and rerun preflight. Broader target coverage belongs in later phase configuration and must be explicitly authorized.
